10 steps for writing an open source project
Aug 27th, 2014

Do you want to publish online some code that you wrote and think that could serve others? Do you know where to start? If not, this guide offers a few steps that you can follow.

Creating a successful project share some similarities with creating a successful company. Sometimes, you have to think like a startup. I can understand some will find this point of view very wrong, because open source is about being free, while a company is about making money. But to live (or “succeed”), an open source project needs users and contributors. And to get these, just like a company, a good product (or good code) is not enough.

By the way, this guide is a bit targeted at PHP projects but definitely applies to any other language.

1. Look for existing projects

Hey, have you seen that “Code” is actually the 2nd step?

This is the #1 thing you should do before even getting started: look around. To get it right, try to answer the following questions:

  • is your project different from the existing ones? if so, how is it better? why should one use it?
  • can you contribute and improve the existing projects instead of creating a new one?

Just to set things clear: there is nothing wrong with wanting to learn, experiment and create our own personal projects. This is not about elitism and restraining developers from creating new projects. But if you want your project to serve and help the community, these questions are essential. Having 10 incomplete and partially maintained projects is worse than 1 or 2 complete, strong and maintained projects.

2. Code

Now, time to code!

3. Follow the standards

Following the standards is not about being a sheep. It’s about making it as easy as possible for visitors and users to grasp how your project work, and wanna-be contributors to make an actual contribution.

PSR

This goes without saying, but your library should work with Composer. Follow PSR-4 (which covers what PSR-0 do) for autoloading and PSR-1 and PSR-2 for coding style.

Directory structure

The following directory structure is the one used in most modern PHP projects:

doc/ # Contains the documentation (if necessary)
src/ # Contains the code
tests/ # Contains the tests

Versioning

I encourage you to follow Semantic Versioning, probably the most popular versioning system. In short, version numbers look like major.minor.patch, e.g. 1.0.0 and you should:

  • increment major version when breaking backward compatibility or adding major features
  • increment minor version when adding features without breaking BC
  • increment patch version number when fixing bugs without breaking BC

The advantage of following that standard is that consumers of the project can confidently upgrade between minor (or patch) versions without worrying about BC breaks.

4. Write some tests

Do you know “innocent until proven guilty”? Well code is broken until proven to work. A project without tests is a project that doesn’t work.

Tests are also essential for getting contributions: how can people contribute to your project if they have no idea if they will break it by doing so?

PHPUnit is the de-facto standard for tests in PHP. If you choose to use any other testing framework, or if there are steps required to get the tests working, you should document it in the README.

Namespacing the tests

Don’t put the tests in the same namespace of the class you test: autocompletion

Use autoload-dev in composer.json.

5. Choose a license

This is a very important step: this is very easy to forget at first, but remember that a project without a license can’t be shared or receive contributions (read more about this).

If you are not sure, the MIT license is a good default, and very popular. Have a look at Choose a License (by GitHub) if you want a good overview.

Remember to put the license in composer.json if this is a PHP project.

6. Publish the code using Git and GitHub

GitHub offers your project great exposure, and Git is today the de-facto standard for code versioning. If you use anything else than Git, it will be much harder for other developers to contribute.

7. Set up continuous integration

Now that you can run your tests locally, set up continuous integration so that they run for every push, and every pull request.

The default option for this is Travis-CI, which is free for open source projects. Once that is done, add the Travis badge to your README to show to visitors that your project is indeed tested and that the build is green.

You can also use several other services that generates reports about the quality of your project, for example:

  • Coveralls generates code coverage reports
  • Scrutinizer-CI gives an overview of the code quality for PHP projects

These services are free for open source projects, and offer badges that you can put in your README.

8. Write some documentation

Every project needs a README. Like for tests, a project without a README is useless.

What a README should contain at least:

  • the project name and a quick description
  • continuous integration badges
  • a “Why?” section that explains why this project exists and what problem it solves
  • a “License” section that states which license the project uses (or you can put that in a LICENSE file)

If you are writing an open source project for learning (i.e. with no intention of maintaining it), that’s very fine, but please make it very clear in the README so that people know what to expect.

If needed, you can add more documentation in separate Markdown files.

If you want to have a website for your project, there are numerous options. I’ll recommend one (shameless self-promotion here): Couscous.

Couscous let’s you write the documentation in Markdown files and put it in the repository, with the code. What it will do is take those Markdown files, turn them into a (customizable) website and publish that website on GitHub pages (i.e. the gh-pages branch). That let’s you have a documentation generator along with a free hosting by GitHub itself. If you want to know more, just check out the website.

9. Sell it

http://www.sideprojectprofit.com/if-you-build-it-they-will-not-come/

OK, don’t actually sell it, but if you went through all this trouble you now want people to use your project. So talk about it. Start with your friends and colleagues, they can give you fast feedback (you know, the kind of stuff you wished you have fixed before sending the URL to the whole planet).

Then share it on Twitter, Reddit is also a great place to get exposure and feedback (here is the PHP subreddit for example).

Keep in mind a very important thing though: if you are asking other people’s valuable time to have a look at your project, make it worthwhile!

  • don’t spam, try to target people that could be interested
  • explain (remember the thing about the README?)
  • but go to the point

In the same spirit, if you are writing Yet Another Router for learning purposes, make it very clear from the beginning.

10. Maintain

That’s it! All the hard work is done. Not!

You have gone through all this, spent so much time writing code, documentation, handling versions and continuous integration and now you have got the attention of a few people. But you’ve done 20% of the work, things really start now :)
